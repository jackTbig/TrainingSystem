------

## 1. 设计原则

### 1.1 建模原则

1. **PostgreSQL 为唯一事务真相源**
2. **核心关系必须关系化建模**
3. **JSONB 仅用于扩展字段，不承载核心业务关系**
4. **课程、题目、文档解析结果必须版本化**
5. **审核、发布、考试必须可追溯**
6. **所有关键实体包含创建时间、更新时间、创建人**
7. **状态字段显式建模，不依赖隐式逻辑**

------

## 2. 命名与通用字段规范

### 2.1 主键规范

- 所有主键统一使用 `UUID`
- 字段名统一为 `id`

### 2.2 时间字段规范

所有核心表包含：

- `created_at TIMESTAMP WITH TIME ZONE`
- `updated_at TIMESTAMP WITH TIME ZONE`

部分表增加：

- `deleted_at TIMESTAMP WITH TIME ZONE`（软删除时）
- `published_at TIMESTAMP WITH TIME ZONE`
- `submitted_at TIMESTAMP WITH TIME ZONE`
- `completed_at TIMESTAMP WITH TIME ZONE`

### 2.3 人员字段规范

常见字段：

- `created_by UUID`
- `updated_by UUID`
- `owner_id UUID`
- `reviewer_id UUID`
- `operator_id UUID`

### 2.4 状态字段规范

统一使用 `VARCHAR(32)` 或枚举映射，状态值在状态机文档中定义。

## 4.1 用户与组织域

### 4.1.1 users

用途：系统用户主表。

关键字段：

- `id`
- `username`
- `password_hash`
- `real_name`
- `email`
- `phone`
- `status`：active / inactive / locked
- `last_login_at`
- `created_at`
- `updated_at`

约束：

- `username` 唯一
- `email` 可唯一，允许为空
- 停用用户不能登录

------

### 4.1.2 roles

用途：角色定义。

关键字段：

- `id`
- `code`：admin / trainer / reviewer / employee
- `name`
- `description`
- `created_at`
- `updated_at`

约束：

- `code` 唯一

------

用途：权限点定义。

关键字段：

- `id`
- `code`
- `name`
- `resource_type`

- `document.upload`

用途：用户角色关联。

关键字段：

说明：

- 可以与各任务专表并存
- 通用表负责统一任务中心展示

### 4.9.2 audit_logs

# 5. 核心关系说明

------

## 5.1 文档与知识点

- 一个文档版本包含多个 chunk
- 一个 chunk 可关联多个知识点
- 一个知识点可关联多个 chunk

关系表：

- `document_knowledge_links`

------

## 5.2 课程与知识点

- 一个课程版本包含多个章节
- 一个章节可映射多个知识点
- 一个知识点可出现在多个课程版本中

关系表：

- `course_knowledge_links`

------

## 5.3 题目与知识点

- 一道题至少关联一个知识点
- 一个知识点可关联多道题

关系表：

- `question_knowledge_links`

------

## 5.4 审核与内容

- 一个课程版本可有多个审核任务
- 一个题目版本可有多个审核任务
- 一次审核任务可有多条评论

------

## 5.5 培训任务与考试

- 一个培训任务可绑定一门课程
- 一个培训任务可绑定一个考试
- 一个用户在一次培训任务中一般对应一次 assignment
- 一个考试可有多次作答 attempt（补考）

------

# 6. 索引建议

------

## 6.1 通用索引

所有外键字段建立普通索引：

- `user_id`
- `document_id`
- `course_id`
- `question_id`
- `training_task_id`
- `exam_id`

------

## 6.2 高频组合索引建议

### documents

- `(status, created_at desc)`
- `(owner_id, created_at desc)`

### knowledge_points

- `(status, parent_id)`
- `name` 模糊检索可配 trigram 或 ES

### course_versions

- `(course_id, version_no desc)`
- `(status, created_at desc)`

### question_versions

- `(question_id, version_no desc)`
- `(status, question_type)`

### review_tasks

- `(assigned_reviewer_id, status, submitted_at desc)`
- `(content_type, content_version_id)`

### training_assignments

- `(user_id, assignment_status, created_at desc)`
- `(training_task_id, user_id)`

### exam_attempts

- `(exam_id, user_id, attempt_no)`
- `(user_id, status, started_at desc)`

### audit_logs

- `(resource_type, resource_id, created_at desc)`
- `(operator_id, created_at desc)`

# 7. 数据一致性原则

1. **发布永远针对 version，不针对主表当前态**
2. **审核永远针对 version**
3. **编辑已提交审核对象时，必须复制新版本**
4. **考试提交接口必须幂等**
5. **异步任务状态更新必须有原子性控制**
6. **ES / Milvus 失败不应破坏主事务，但需可补偿**
